大数据组件

Posted on 2020-08-25 Edited on 2021-03-28

HDFS

Hadoop分布式存储文件系统。容错高，冗余存储、冗余复制。

冗余因子默认为3，其中有2个副本放在同一机架的不同节点。放置位置随机。

当某一节点存储的副本出现错误，会从其他节点复制一份。可以查错。

HBase:

列式存储，可以基于HDFS或普通文件系统存储。HBase表视图类似关系数据库的表，但是物理层面是列式存储的。以列族为单位存放。

完整列名表述 列族名 ：列限定符

可以通过Hadoop MapReduce方法生成索引表。

HBase通过[行键，列族，列限定符，时间戳]确定一个值，也可视为Key-Value数据库。

删除操作会保留旧版本，通过时间戳进行区分。

不支持事务，无法实现跨行的原子性。无法使用SQL进行访问（Hive下可以提供类似SQL语言的方法访问HBase）。

有些地方可能是空值，因此HBase是稀疏的。

Region：HBase中根据rowKey的范围进行切分，形成一个分区Region，然后分发到不同的Region Server，当Region存储内容的大小超过阈值时，可以分裂成两个Region。一个Region不能存储在两台服务器上。

HLog：系统将MemStore中的数据写入 磁盘的StoreFile后，在HLog写入标记表示已写入磁盘，并清除缓存。每个Region Server维护一个HLog。

META表：HBase中一张特殊的数据表，存储了所有的Region的位置信息，META 表本身存储在ZooKeeper上。

Store：Region Server的核心，每个Store对应表中一个列族的存储，每一个Store包含一个MemStore缓存和若干个StoreFile文件。

HDFS是Hadoop分布式文件系统，为HBase提供了高可靠性的底层存储支持。

HBase是Hadoop DataBase。HBase在HDFS之上提供了：

1、高并发实时随机写。通过 LSM（内存+顺序写磁盘）的方式提供了 HDFS 所不拥有的实时随机写及修改功能。 2、高并发实时读。

在业务层面上，HBase可以独立于HDFS来理解。

ZooKeeper

集群管理

Hive

构建在Hadoop上的数据仓库，提供类sql（hql）对非关系型数据库进行访问，查询语句转化为MapReduce作业并提交到Hadoop上运行。

MapReduce

编程框架

Map Shuffle Reduce

大文件切片，输入到不同的机器上执行Map任务。集群中一台机器为Master（协调调度任务），其他机器为Worker（执行Map/Reduce任务）

Map阶段：输入<key, value> ，输出<key, 1>

Shuffle阶段（在Map端完成）：排序，分区，得到<key, value-list> 若没有定义Combiner函数，Shuffle会将相同key的键值对进行Merge，形成<key, 1-1-1-…> 自定义Combiner函数可以先求和，形成<key,value>

Reduce阶段：汇总计算，得到<key, value>

Pig

处理大规模数据的脚本语言，用户只需编写几条简单的语句，就能转化成MapReduce作业。

解决了Hadoop中抽象层次低、需编写大量代码的问题。

Pig Latin语言，类似SQL。自动优化。

Pig通常用于ETL过程，数据采集后通过Pig进行统一加工处理，然后加载到数据仓库Hive中。

Pig面向批处理。

作业调度

Ooize

工作流和协作引擎，协调Hadoop上运行不同的作业。

解决了Hadoop中处理作业依赖关系的问题。

AzKaban

与Ooize一样 都是广泛使用的工作流调度程序。

二者相同之处：

两者均可以调度 Linux 命令、MapReduce、Spark、Pig、Java、Hive 等工作流任务；

两者均可以定时执行工作流任务。

AzKaban使用 Properties(Flow 1.0) 和 YAML(Flow 2.0) 文件定义工作流

Oozie 使用 Hadoop 流程定义语言（hadoop process defination language，HPDL）来描述工作流，HPDL 是一种 XML 流程定义语言。

Azkaban 有较严格的权限控制，如用户对工作流进行读/写/执行等操作；

Oozie 暂无严格的权限控制。

Tez

支持DAG（有向无环图）作业的计算框架，对作业操作进行重新分解和组合，形成一个大的DAG作业，减少不必要操作。

解决了Hadoop中不同的MapReduce任务存在重复操作的问题。

源于MapReduce框架，核心思想：将Map和Reduce操作继续拆分。

Map：Input、Processor、Sort、Merge、Output

Reduce：Input、Shuffle、Sort、Merge、Processor、Output

Tez将这些操作自由组合，形成大的DAG作业。

Kafka

    消息队列，数据交换

YARN：

MapReduce1.0使用Master/Slave架构设计，对应一个JobTracker（负责作业调度和资源管理）和若干个TaskTracker（负责执行JobTracker指派的任务）。存在问题：JobTracker任务过重，TaskTracker容易内存溢出（CPU/内存的资源分配不根据实际情况进行），Map和Reduce资源分配不合理导致效率低下。

YARN实际上是将MapReduce1.0中的资源管理调度功能抽取出来的框架，此后MapReduce2.0就是纯计算框架,YARN负责调度。

YARN三大组件：ResourceManager，ApplicationMaster，NodeManager

YARN工作原理简述：

YARN中使用容器（Container）而非槽（Slot）。

YARN中的ApplicationMaster可以面向不同的计算框架，除了MapReduce还有Storm和Spark。从这可以了解YARN最终可实现“一个集群多个框架”。

与YARN类似的资源调度框架还有Mesos、Torca、Corona和Borg。

Spark

Spark Core

RDD 弹性分布式数据集/只读分区记录集合。

窄依赖：一个父RDD的分区对应一个子RDD分区，可以流水线的方式计算父分区。典型操作：map、filter、union

宽依赖，一个父RDD的分区对应多个子RDD分区，通常伴随Shuffle操作。典型操作：groupByKey、sortByKey

在DAG中进行反向解析，遇到宽依赖断开，遇到窄依赖则划分到同一阶段，加入流水线。

闭包：在创建函数时，如果需要捕获自由变量，那么包含指向被捕获变量的引用的函数就被称为闭包函数。

Spark SQL

DataFrame：用于结构化数据，有明确的行和列， RDD：用于非结构化数据。DataFrame可以理解为带有Schema的RDD。

Dataset：分布式数据集合，结合了RDD和DataFrame的优点

Spark Streaming

将流数据分解成一系列短小的批处理作业，使用Spark Core进行批处理计算，变相实时处理。无法实现毫秒级实时响应（Storm可以）。

Spark Streaming实现了至少消费一次、只消费一次的语义。Flink也实现了，Storm没有。

Flume

Apache Flume 分布式、高可用的数据收集系统，将大规模流数据导入HDFS的工具。

Phoenix

Kudu

与HBase类似的列式存储分布式数据库。

为什么有了HDFS和HBase还需要Kudu？

HDFS适合离线分析，不适合单条记录的update操作，随机读写性能差。

HBase可以进行高效随机读写，不适于基于SQL的数据分析，大批量数据获取时性能差。

Kudu较好地解决了上述二者的缺点。

Sqoop

Sqoop 是一个常用的数据迁移工具，主要用于在不同存储系统之间实现数据的导入与导出.

导入数据：从 MySQL，Oracle 等关系型数据库中导入数据到 HDFS、Hive、HBase 等分布式文件存储系统中;

导出数据：从 分布式文件系统中导出数据到关系数据库中。 

其原理是将执行命令转化成 MapReduce 作业来实现数据的迁移。

Apache Sqoop是为了将数据从结构化存储设备批量导入HDFS设计，如关系型数据库。

应用场景：白天生产的数据库中的数据在晚上导入Hive数据仓库。

Bulkload

HBase导入大量数据。

ACL

大数据场景下高并发分布式访问控制。

Sentry

Sentry是一个实时事件日志记录和聚合平台。它专门用于监视错误和提取执行适当的事后操作所需的所有信息,而无需使用标准用户反馈循环的任何麻烦。

Impala

Impala是Cloudera公司推出，提供对HDFS、HBase数据的高性能、低延迟的交互式SQL查询功能。

特点：

基于内存进行计算，能对PB级别的数据进行交互式实时查询。（相应的，对内存依赖大）

无需转换为MapReduce，直接读取HBase/HDFS数据，降低延迟。

C++编写，LLVM统一编译运行。（开源？）

具有数据仓库的特性，可直接对Hive数据做数据分析。

支持Data Local，列式存储，兼容HiveSql，支持JDBC/ODBC远程访问。
